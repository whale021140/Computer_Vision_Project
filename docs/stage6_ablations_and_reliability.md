# Stage 6：机制消融与可靠性验证

> 状态：重新设计完成，尚未开始运行  
> 主结果基线：Stage 5.6 `v2_wide` 最终宽范围校准结果  
> 环境：`ece485`

## 1. Stage 6 的定位

Stage 5.6 已回答主问题：在完全相同的冻结检测器候选、few-shot
划分、轻量预测头训练协议和开发集选择规则下，比较 CLIP、
CLIP+DINOv2 与 SigLIP 2。

Stage 6 不再重新搜索一个“更好看的主表”，而是回答主结果背后的四个
机制问题：

1. 性能提升究竟有多少来自显式基数预测，而不是普通候选匹配？
2. 空间坐标、相似度特征和层级计数结构分别贡献多少？
3. 3+ 样本仍然困难的原因是候选缺失、重复框、排序错误还是计数错误？
4. 在反事实表达中，模型能否拒绝语义上不存在的目标？

Stage 5.6 v2 仍是最终主实验。Stage 6 是预先定义的消融和可靠性分析，
结果无论正负都报告。

## 2. 防止再次发生测试驱动选择

Stage 6 使用三级门控：

1. **开发集探索**：只在 Stage 5.6 图像不重叠 dev 上运行所有 pilot、
   选择结构、选择 epoch 和校准。
2. **锁定确认集合**：在看到新的 testA/testB 结果前，把要保留的对照、
   checkpoint、校准范围、指标和停止规则写入 manifest。
3. **单次测试确认**：只有全部锁定对照通过完整性审计后，才一次性运行
   testA/testB。测试结果不再触发结构修改、重训或重新校准。

项目早期已经看过 testA/testB，因此报告只能准确声称“Stage 6 的选择
不使用 Stage 6 test 指标”，不能声称测试集在整个项目生命周期从未被
观察。

## 3. Stage 6.0：冻结基线与审计

开始任何新训练前固定以下基线：

- Stage 5.6 统一 train/dev/test 划分及哈希；
- hierarchical 胜出方案；
- 每个单元的开发集所选 checkpoint；
- `v2_wide` 最终校准网格、无边界审计及主结果；
- 相同 Faster R-CNN 候选和冻结特征缓存；
- 主选择指标为四个数量组等权的 count-macro mean F1；
- 辅助指标为官方 F1、`T_acc`、`N_acc`、exact-set、cardinality
  accuracy，以及按 0/1/2/3+ 分组的结果。

这一阶段只做读取和审计，不训练。

冻结的 10% 主基线如下；数值为三个种子的均值 ± 样本标准差：

| 测试集 | CLIP F1 | CLIP+DINOv2 F1 | SigLIP 2 F1 |
|---|---:|---:|---:|
| testA | 0.3605 ± 0.0028 | 0.3438 ± 0.0103 | **0.4625 ± 0.0107** |
| testB | 0.3430 ± 0.0041 | 0.3288 ± 0.0065 | **0.4180 ± 0.0044** |

Stage 5.6 的 1%→5%→10% 官方 F1 与 mean set F1 在全部六条
表示/测试曲线上均单调上升；SigLIP 2 在六个表示比较中全部最高，
CLIP+DINOv2 在六个比较中全部低于 CLIP。Stage 6 不以打破这些数值为
完成条件，而以解释其来源和限制为完成条件。

## 4. Stage 6.1：基数机制核心消融

### 4.1 开发集 pilot

先用 `SigLIP 2, 10%, seed 0` 比较：

| 编号 | 方案 | 要回答的问题 |
|---|---|---|
| C0 | membership-only | 不使用基数头，仅按 membership 阈值输出集合时会怎样？ |
| C1 | flat cardinality | 原始平坦 0/1/2/3+ 分类头是否足够？ |
| C2 | hierarchical | Stage 5.6 胜出结构，作为基线 |
| C3 | hierarchical without count gating | 训练基数头但推理不让它控制集合大小，收益来自训练还是决策？ |
| C4 | count gating without logit calibration | 去掉 class-0/class-3 bias 后损失多少？ |

`membership-only` 必须允许空集合，不能强制至少输出一个框，否则它既不
是公平消融，也不符合 GREC。

### 4.2 基数损失权重

在相同 hierarchical 结构上只用开发集比较：

```text
lambda_cardinality = {0, 0.25, 0.5, 1.0, 2.0}
```

其中 `lambda=0` 是“保留结构但不学习计数”的严格控制组，不是当前主
模型设置。每个值使用相同训练预算、epoch 选择规则和最终宽校准门。

### 4.3 确认范围

pilot 结束后只保留以下预先规定的关键对照进行三种子确认：

- C0 membership-only；
- C1 flat cardinality；
- C2 hierarchical；
- 开发集选出的非默认 `lambda`（若与 1.0 相同则不增加重复单元）。

核心确认使用 `SigLIP 2, 10%, seeds 0/1/2`。这样可量化基数机制，避免
把所有消融无必要地扩成新的 27 单元大网格。

### 4.4 校准稳定性审计

Stage 5.6 已消除搜索边界截断，但同一个 dev 同时承担 recipe、epoch 和
校准选择。Stage 6 因此增加只读稳定性分析，不改写 Stage 5.6 主结果：

- 对 dev 按图像做 bootstrap，报告所选 bias/threshold 与四组 mean F1
  的分布；
- 在互斥 dev 图像子集上重复校准，检查设置是否剧烈漂移；
- 比较无校准、`v2_wide` bias 校准和更低自由度的 temperature/prior
  correction；
- 画 `N_acc`、`T_acc`、官方 F1 和 count-macro mean F1 的工作点曲线；
- 所有分析只用于解释稳定性，不根据 testA/B 反向挑选校准方法。

## 5. Stage 6.2：输入信息与参数量消融

### 5.1 空间信息

在同一基线上比较：

- 完整输入；
- 去掉归一化框坐标；
- 只保留框坐标与显式图文相似度；
- 去掉显式相似度，仅保留冻结图像/文本特征。

它们用于检验 proposal 中“坐标弥补 CLIP 空间推理不足”的动机。

### 5.2 参数匹配表征比较

当前三种表示的输入维度不同，轻量头参数量也不同。增加一个参数匹配的
辅助比较：

1. 只在训练池上、无标签地拟合固定 PCA；
2. 将候选视觉特征和文本特征分别映射到共同维度；
3. CLIP 使用等价的 512 维通路，另外两种表示也固定到相同维度；
4. PCA 不参与监督训练，三个预测头拥有相同可训练结构和参数量；
5. 使用 10%、三种子比较三种表征。

该表是 Stage 5.6 表征结论的控制实验，不覆盖原始特征的主表。报告必须
同时给出降维后的保留方差，防止把某一表示因降维受损误写成表示本身
更差。

## 6. Stage 6.3：3+ 与候选框诊断

### 6.1 更细的真实数量分组

将原来的 `3+` 拆成：

```text
3, 4, 5, 6+
```

每组报告样本数、proposal full-target coverage、mean F1、exact-set 和
cardinality accuracy。样本过少的组只作描述，不作强统计结论。

### 6.2 错误来源分解

每条 3+ 样本归入互斥的首要失败原因：

1. proposal miss：候选池没有覆盖全部真实框；
2. count under/over-estimation：预测数量错误；
3. duplicate selection：多个高度重叠候选重复对应同一目标；
4. ranking/selection error：候选存在、数量也足够，但选择了错误区域；
5. success。

这比单独报告一个 3+ F1 更能指导改进。

### 6.3 候选上限与重复抑制

先在固定开发集图像上复用检测器原始输出，比较：

```text
max_proposals = {50, 100, 150, 200}
NMS threshold = {0.5, 0.7, 0.85}
```

先报告 proposal recall、平均候选数、达到上限的图像比例和运行开销。
只有开发集诊断显示有实际收益时，才选择一个替代设置训练三种子确认。

另外比较推理期去重策略：

- 无额外去重；
- 类别无关 NMS；
- 对已选框做目标级 greedy suppression。

Stage 5.5 的 one-to-one 正样本标签结果作为历史对照，不重复宣称为新
实验。

## 7. Stage 6.4：反事实可靠性

proposal 中把 C-RefCOCO、C-RefCOCO+、C-RefCOCOg 列为“预处理可兼容
时”的可靠性测试，因此按以下顺序执行：

1. 建立表达、图像和框格式转换器；
2. 审计图像可用性、表达数量、目标定义及与 gRefCOCO 指标的兼容性；
3. 冻结 Stage 5.6 的 10% 三种子模型，不在反事实数据上训练或校准；
4. 报告反事实拒绝率、错误 grounding 率，以及原表达到反事实表达的
   置信度变化；
5. 三个数据集若有任何一个不兼容，明确记录原因和缺失字段，不静默
   删除。

FineCops-Ref 保持 optional。它只有在前三个反事实数据集完成且时间允许
时进入 Stage 6.4b。

## 8. Stage 6.5：定性结果与外部方法背景

对 Stage 5.6 v2 的 10% SigLIP 2 和 CLIP 生成统一定性图，至少覆盖：

- 正确拒绝 no-target；
- single-target 正确与混淆案例；
- 双目标完整与漏检案例；
- 3+ 的 proposal miss、重复选择和计数失败；
- SigLIP 2 相对 CLIP 的代表性胜例和负例。

RECANTFormer、HieA2G 等大型专用 GREC 方法只放入“文献背景”表。除非能
获得兼容 checkpoint 并严格复现相同数据/指标，否则不把论文数字与本
项目 few-shot 结果写成直接公平排名。RegionCLIP 仍是 proposal 中的
时间允许扩展，可作为 Stage 6.x，而不是 Stage 6 核心完成条件。

## 9. 运行规模与停止规则

为控制算力：

- 机制 pilot：单表示、单比例、单种子；
- 关键机制确认：10% SigLIP 2、三种子；
- 参数匹配表征确认：三种表示、10%、三种子；
- 候选参数只先做无需训练的 proposal/dev 诊断；
- 只有开发集预声明门通过的候选改动才训练确认；
- 不再完整复制 1%/5%/10% × 三表示 × 三种子的主网格。

任一校准轴命中边界时，在访问 Stage 6 test 前停止并只用 dev 扩展；
所有数值必须有限；所有三种子单元必须齐全；任何缺失或不兼容都写入
manifest。
